# Fennec EMF OSGi API

The core API module for Eclipse Fennec EMF OSGi. Defines all interfaces, annotations, constants, and utilities needed to integrate the Eclipse Modeling Framework (EMF) with OSGi's dynamic service model.

## Overview

This module provides:

- **Service interfaces** for OSGi-based EMF resource management (`ResourceSetFactory`, `EPackageConfigurator`, `ResourceSetConfigurator`)
- **Annotations** for declaring EMF model capabilities and requirements in OSGi bundles
- **Constants** (`EMFNamespaces`) for consistent service property naming across the framework
- **Utilities** for loading ecore models (`EcoreHelper`), managing service properties (`ServicePropertyContext`), and delegating EMF registries
- **Delegating registries** that bridge EMF's static registry model with OSGi's dynamic services

All types are exported as versioned OSGi packages.

## Core Interfaces

### ResourceSetFactory

The primary entry point for consumers. Provides fully configured `ResourceSet` instances with all registered `EPackage`s and `Resource.Factory`s pre-configured.

```java
@ProviderType
public interface ResourceSetFactory {
    ResourceSet createResourceSet();
}
```

**Usage in DS components:**
```java
@Reference
ResourceSetFactory rsFactory;

void doWork() {
    ResourceSet rs = rsFactory.createResourceSet();
    // rs is pre-configured with all available EPackages and ResourceFactories
}
```

**Filtering by model:**
```java
@Reference(target = "(emf.name=mymodel)")
ResourceSetFactory rsFactory;
```

### EPackageConfigurator

Registers/unregisters an `EPackage` in an EMF `EPackage.Registry`. The framework tracks these services via the whiteboard pattern.

```java
@ProviderType
public interface EPackageConfigurator {
    void configureEPackage(EPackage.Registry registry);
    void unconfigureEPackage(EPackage.Registry registry);
}
```

Implementations are typically generated by the code generator or created by the model extender. Each `EPackageConfigurator` service carries properties like `emf.name`, `emf.nsURI`, and `emf.model.scope`.

### ResourceSetConfigurator

Applies additional configuration to a `ResourceSet` after creation. Multiple configurators are applied by the `ResourceSetFactory`.

```java
@ProviderType
public interface ResourceSetConfigurator {
    void configureResourceSet(ResourceSet resourceSet);
}
```

### Other Interfaces

| Interface | Purpose |
|-----------|---------|
| `AutoClosableResourceSet` | Combines `ResourceSet` + `AutoCloseable` for try-with-resources |
| `Detachable` | Detaches `EObject`s from their `Resource` and/or `ResourceSet` |
| `HughDataResourceSet` | Specialized `ResourceSet` for large datasets with `ResourceLocator` cache and notification suppression |
| `ResourceSetCache` | Provides a single cached `ResourceSet` instance (non-thread-safe) |
| `RegistryPropertyListener` | Callback for registry service property changes |
| `RegistryTrackingService` | Tracks EMF registry services and dispatches property-change events |
| `UriHandlerProvider` | Supplies custom `URIHandler` implementations |
| `UriMapProvider` | Provides URI-to-URI redirection maps |

## Annotations

### Model Declaration

**`@EMFModel`** -- Component property type annotation for DS components. Declares EMF model metadata as service properties:

```java
@Component
@EMFModel(name = "mymodel", nsURI = "http://example.org/mymodel/1.0", version = "1.0.0")
public class MyModelConfigurator implements EPackageConfigurator { ... }
```

Generated service properties: `emf.name`, `emf.nsURI`, `emf.contentType`, `emf.fileExtension`, `emf.protocol`, `emf.feature`, `emf.version`.

### Capability/Requirement Annotations

| Annotation | Namespace | Purpose |
|-----------|-----------|---------|
| `@RequireEMF` | `emf.core` | Requires the Fennec EMF runtime |
| `@ProvideExtenderModel` | `osgi.extender` | Requires the EMF model extender; specifies model paths |
| `@EPackage` | `org.eclipse.emf.ecore.generated_package` | Declares a generated EPackage capability |
| `@DynamicEPackage` | `org.eclipse.emf.ecore.dynamic_package` | Declares a dynamically loaded EPackage capability |
| `@EMFConfigurator` | `emf.configurator` | Marks and declares a configurator capability |

### Extender Annotation

Use `@ProvideExtenderModel` to register ecore models from bundle resources without code generation:

```java
@ProvideExtenderModel({"/model", "OSGI-INF/ecore/special.ecore"})
package com.example.mymodel;

import org.eclipse.fennec.emf.osgi.annotation.extender.ProvideExtenderModel;
```

## EcoreHelper

Utility class for loading EMF ecore models from various sources. Supports both instance-based usage (with a shared `ResourceSet`) and static one-shot loading.

### Quick Start

```java
// One-shot: load and detach (no resource management needed)
EcoreHelper helper = new EcoreHelper();
EPackage ePackage = helper.loadEcoreDetached("mymodel.ecore", getClass());

// Attached: EPackage stays in ResourceSet, diagnostics accessible
EcoreHelper helper = new EcoreHelper();
EPackage ePackage = helper.loadEcore("mymodel.ecore", getClass());
// Access load diagnostics:
ePackage.eResource().getErrors();
ePackage.eResource().getWarnings();
// Cleanup when done:
helper.releaseAll();
```

### Loading Modes

| Mode | Method prefix | Resource after load | Use case |
|------|--------------|-------------------|----------|
| **Attached** | `loadEcore(...)` | EPackage remains in Resource, URI set to nsURI | When you need diagnostics or further Resource operations |
| **Detached** | `loadEcoreDetached(...)` | EPackage removed from Resource | Lightweight, standalone usage, tests |
| **Generic** | `load(..., Class<T>)` | Attached, returns typed EObject | Loading any EObject type (EPackage, EClass, etc.) |

### Loading Sources

All three modes support loading from:

- `InputStream` + `URI` -- raw stream with explicit URI
- `URL` -- file:// or bundle:// URL
- `Path` -- filesystem path
- `String` + `Class<?>` -- classpath resource relative to a context class

### Static Convenience

```java
// Pre-configured ResourceSet with EcorePackage, XMI factories
ResourceSet rs = EcoreHelper.createResourceSet();

// One-shot load with provided ResourceSet
EPackage pkg = EcoreHelper.loadEcore(url, resourceSet);

// Lookup helpers
EClass cls = EcoreHelper.getEClass(ePackage, "MyClass");
EStructuralFeature feat = EcoreHelper.getFeature(cls, "myAttribute");

// Path property extraction: "/model;foo=bar;key" -> "/model" + {foo=bar, key=null}
Map<String, String> props = new HashMap<>();
String path = EcoreHelper.extractProperties("/model;foo=bar;key", props);
```

## Service Property Context

The `ServicePropertyContext` mechanism aggregates service properties from multiple OSGi services, enabling dynamic property propagation through the registry whiteboard pattern.

```java
ServicePropertyContext ctx = ServicePropertyContext.create();

// Add properties from service registrations (keyed by service.id)
ctx.addSubContext(serviceProperties1);
ctx.addSubContext(serviceProperties2);

// Get merged properties for re-registration
Dictionary<String, Object> merged = ctx.getDictionary(true);
```

This is used internally by registry components to dynamically update their service properties when configurators are added or removed.

## Delegating Registries

### DelegatingEPackageRegistry

Extends `EPackageRegistryImpl` to combine a local registry with a delegate. Used to layer ResourceSet-level registrations on top of the global registry.

### DelegatingResourceFactoryRegistry

Extends `ResourceFactoryRegistryImpl` to delegate resource factory lookup to a parent registry. Supports `MapChangeListener` for reactive updates when factories are added/removed.

## Testing

```bash
# Unit tests only
./gradlew :org.eclipse.fennec.emf.osgi.api:test

# Full build (required when API changes affect downstream modules)
./gradlew build
```

Unit tests cover `EcoreHelper` comprehensively: all loading modes, all sources, error handling, resource lifecycle, property extraction, and edge cases (null nsURI, empty streams, type mismatches).
